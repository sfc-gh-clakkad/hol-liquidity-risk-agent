Exercise 2: LCR Calculation Pipeline

Translate LCR domain requirements into Snowpark Python stored procedures -- CoCo generates production-ready calculation logic from business context.

35 minutes
4 Tasks

Deliverables

  • RUN_LCR_FORECAST() -- Snowpark stored procedure computing a 150-day LCR forecast
  • RUN_WHAT_IF_SCENARIO(WHAT_IF_ID) -- Snowpark stored procedure for stress scenarios
  • Both procedures tested and producing correct results in PRESENTATION tables

"I want to say 'recalculate LCR' in the chat and have it happen. No more waiting."

James Morrison, Head of Treasury Risk
1Understand the Calculation Requirements
PromptBased on the domain context already loaded, explain the LCR calculation pipeline step by step: 1. How does HQLA projection work? (haircuts, decay, maturity) 2. How are cashflows projected over 150 days? 3. How is netting done (inflows vs outflows)? 4. How is the final LCR ratio computed? 5. How does the what-if process differ from production? Keep it concise -- bullet points for each step.
2Generate the LCR Forecast Procedure

This is where CoCo's power shines -- translating domain knowledge into working Snowpark code.

PromptCreate and deploy a Snowpark Python stored procedure called RUN_LCR_FORECAST in LIQUIDITY_RISK_DB.PUBLIC that computes a 150-day LCR forecast. Use the domain context and table schemas already loaded. Proceed autonomously. Requirements: - LANGUAGE PYTHON, RUNTIME_VERSION 3.11, EXECUTE AS CALLER - Packages: snowflake-snowpark-python - Takes no parameters, returns VARCHAR Pipeline steps to implement: 1. HQLA Projection (haircuts, decay, maturity zeroing) 2. HQLA Summary (unpivot wide to long, aggregate per day) 3. Cashflow Projections (inflows + outflows over 150 days) 4. Netting (net = inflows + outflows, compute ABS) 5. Final LCR (HQLA / net_cash_outflows, INSERT to PRESENTATION.LCR) 6. Return informative success message Deploy the procedure to Snowflake after generating it.
What's happening: CoCo's Snowpark Python skill activates automatically -- ensuring correct HANDLER signatures, RUNTIME_VERSION, PACKAGES declaration, and EXECUTE AS CALLER for agent compatibility.
3Generate the What-If Scenario Procedure
PromptCreate and deploy a Snowpark Python stored procedure called RUN_WHAT_IF_SCENARIO in LIQUIDITY_RISK_DB.PUBLIC that executes a what-if stress scenario. Proceed autonomously. Requirements: - Takes WHAT_IF_ID (VARCHAR) as parameter, returns VARCHAR - Clones sandbox tables, applies stress factors via MERGE - Runs same LCR pipeline on stressed data - Writes results to PRESENTATION.WHAT_IF_LCR - Cleans up temp tables after Deploy the procedure to Snowflake after generating it.
4Test the Procedures
PromptCall the RUN_LCR_FORECAST procedure and show me the results. Proceed autonomously. 1. CALL LIQUIDITY_RISK_DB.PUBLIC.RUN_LCR_FORECAST(); 2. Query results for days 1, 7, 14, 30, 60, 90, 150 3. Is Day 1 LCR above 1.0 (Basel III compliant)? 4. Show the trend -- improving or declining?
PromptCall the RUN_WHAT_IF_SCENARIO procedure with scenario 1 (Market Stress) and compare results to baseline. Proceed autonomously. 1. CALL LIQUIDITY_RISK_DB.PUBLIC.RUN_WHAT_IF_SCENARIO('1'); 2. Compare baseline vs stressed LCR for key days 3. The stressed LCR should be lower than baseline -- is this the case?

Validation Checklist

  • RUN_LCR_FORECAST() exists in LIQUIDITY_RISK_DB.PUBLIC
  • RUN_WHAT_IF_SCENARIO(VARCHAR) exists in LIQUIDITY_RISK_DB.PUBLIC
  • PRESENTATION.LCR populated with 150-day forecast
  • Day 1 LCR above 1.0 (Basel III compliant)
  • What-if scenario 1 produces results in WHAT_IF_LCR
  • Stressed LCR is lower than baseline (stress impact visible)
  • Both procedures return informative success messages
  • Procedures use EXECUTE AS CALLER (for agent invocation)

3Create the Multi-Tool Agent

The centerpiece -- three tools in one agent:

ToolTypePurpose
LIQUIDITY_ANALYSTcortex_analystNatural language queries via semantic view
RECALCULATE_LCRstored_procedureTriggers full LCR forecast recalculation
RUN_WHAT_IFstored_procedureExecutes a what-if stress scenario
PromptCreate a Cortex Agent called LIQUIDITY_FORECAST_AGENT in LIQUIDITY_RISK_DB.PUBLIC with display name "Liquidity Risk Agent". Proceed autonomously. THREE tools: 1. LIQUIDITY_ANALYST - queries via LIQUIDITY_SV semantic view 2. RECALCULATE_LCR - calls RUN_LCR_FORECAST procedure 3. RUN_WHAT_IF - calls RUN_WHAT_IF_SCENARIO procedure Orchestration rules: - Data questions -> LIQUIDITY_ANALYST - Recalculation requests -> RECALCULATE_LCR, then LIQUIDITY_ANALYST - What-if requests -> RUN_WHAT_IF, then LIQUIDITY_ANALYST - Flag any LCR below 1.0 as regulatory concerns Deploy the agent to Snowflake.

Verify: SHOW AGENTS IN SNOWFLAKE INTELLIGENCE SNOWFLAKE_INTELLIGENCE_OBJECT_DEFAULT;
